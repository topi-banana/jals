# Rhai build script example

This project runs `build.rhai` before `javac`. The script uses only JALS's storage-backed API to
generate `target/jals/build/rhai/out/generated/BuildInfo.java`, adds it as a source, and contributes
compiler/JVM flags and subprocess environment entries.

## Run it

From this directory:

```sh
cargo run -p jals-cli -- run
```

With `jals` installed, use `jals run`. The program prints:

```text
generated by build.rhai
rhai
rhai
```

`jals build --dry-run` still runs the script so the displayed `javac` command includes its generated
source and `-Xlint:all`. It publishes managed output and cache state under `target/jals`, but it
does not run `javac` and does not apply exclusive source-tree publications — a preview never
rewrites or deletes anything outside `target/jals`. `jals clean` removes `target/classes` and
`target/jals/build`, including generated script output.

## What it demonstrates

- `[build] script = { type = "rhai", file = "build.rhai" }` enables the optional pre-build phase.
- `project.exists` reads the immutable project snapshot; `output.write_text` can publish only below
  `target/jals/build/rhai/out`.
- `build.add_source`, `add_javac_arg`, `add_jvm_arg`, `set_compile_env`, and `set_run_env` augment the
  later compile/run requests.
- `rerun_if_changed` and `rerun_if_env_changed` narrow cache invalidation to declared inputs.
- `warning` reports a non-fatal diagnostic, while `error` prevents all pending output from being
  published. `metadata` returns host-readable key/value data without changing `javac` or `java`.

See [`jals-build/README.md`](../../jals-build/README.md#rhai-build-scripts) for the complete API and
sandbox/cache behavior.
